# ACM BYOK Tooling HOWTO

This directory contains the necessary Containerfiles to build and use the ACM Bring Your Own Knowledge (BYOK) tool. The tool allows you to encapsulate your custom Markdown documents into an AI-ready vector database container image, leveraging the same embedding processor as the core ACM RAG content.

## Design

The purpose of the tool image is to encapsulate everything needed to build a BYOK container image in a single environment. 
The tool image wraps `buildah` and the ACM RAG embeddings generator (`custom_processor.py`) along with its Python dependencies managed by `uv`. 

There are two mandatory parameters when running the tool: 
- An input directory with your Markdown content (`/markdown`)
- An output directory for the resulting container image tarball (`/output`)

## How to build the BYOK tool image locally

You can build the tool image using the included Makefile target:

```bash
make build-byok-tool
```

Or manually:

```bash
$ MY_BYOK_TOOL_IMAGE=localhost/acm-rag-tool:latest
$ podman build --build-arg BYOK_TOOL_IMAGE=$MY_BYOK_TOOL_IMAGE -t $MY_BYOK_TOOL_IMAGE -f byok/Containerfile.tool .
```

## How to run the BYOK tool

Once the tool image is built, you can run it to convert a local directory of Markdown files into a vector database container image. 

Using the Makefile:

```bash
make run-byok-tool DOCS_FOLDER=docs/acm/2.15 OUTPUT_DIR=output_image BYOK_OUT_IMAGE_TAG=my-custom-acm-byok
```

Or manually:

```bash
$ MY_BYOK_TOOL_IMAGE=localhost/acm-rag-tool:latest
$ podman run -e OUT_IMAGE_TAG=acm-byok-image -e DOCS_VERSION=2.15 -it --rm --device=/dev/fuse \
  -v /path/to/your/markdown_dir:/markdown:Z \
  -v /path/to/your/output_dir:/output:Z \
  $MY_BYOK_TOOL_IMAGE
```

**Note for Docker Users:** If you are using Docker instead of Podman, the internal container builder (`buildah`) requires running with elevated privileges. Replace `--device=/dev/fuse` with `--privileged`:

> ⚠️ **SECURITY WARNING:** Running containers in `--privileged` mode grants them nearly full access to the host machine. While this is necessary for Docker to run `buildah` (Docker-in-Docker), it is a significant security risk. Do not run this tool via Docker in shared, multi-tenant, or production CI environments unless strictly sandboxed. Podman (which uses `--device=/dev/fuse` for rootless building) is the strongly recommended and safer alternative.

```bash
$ docker run -e OUT_IMAGE_TAG=acm-byok-image -e DOCS_VERSION=2.15 -it --rm --privileged \
  -v /path/to/your/markdown_dir:/markdown:Z \
  -v /path/to/your/output_dir:/output:Z \
  $MY_BYOK_TOOL_IMAGE
```

There are two mandatory mounted directories:
- `/markdown` is the root of the directory tree containing the content to be included in the BYOK RAG database, in Markdown. File extensions are expected to be `.md`.
- `/output` is the directory where the resulting image tar archive will be written.

The `OUT_IMAGE_TAG` environment variable can be used to override the tag of the generated image. It defaults to "byok-image".
The `VECTOR_DB_INDEX` environment variable can be used to override the Faiss database index ID. It defaults to "vector_db_index".
The `DOCS_VERSION` environment variable can be used to control the documentation version embedded in the URL generated by the `custom_processor.py`. It defaults to "byok".

The BYOK tool will produce the resulting container image as a tar archive named `<dir_for_image_tar>/acm-byok-image.tar`.

The generated image can be instantiated like so:

```bash
$ podman load < /path/to/your/output_dir/acm-byok-image.tar
$ podman images
REPOSITORY                      TAG                IMAGE ID      CREATED         SIZE
localhost/acm-byok-image        latest             ...           ...             ...
```

You can then tag and push this generated image to a registry.

## Fast Local Builds

For rapid development and local testing, creating the intermediate tarball via `buildah` can add unnecessary overhead. If you have the Python toolchain (`uv`) installed on your local host, you can generate the vector database locally and wrap it into a container image directly.

You can trigger this using:

```bash
make build-byok-fast
```

This will run the embedding generator via `uv` natively on your machine, then run a standard `podman build` (or `docker build`) to package the resulting folder directly into `localhost/acm-byok-image:latest` in your local container daemon.

To push the image generated this way to a registry:

```bash
make push-byok-fast BYOK_REGISTRY=quay.io/myuser
```

## Document Metadata

The BYOK tool supports metadata parsing to populate the RAG database with document titles and URLs. This is handled by `embedding_generator/custom_processor.py`.

It supports standard YAML frontmatter in Markdown files or custom header mapping files (`header_map.json`) as described in the root repository documentation.

## OpenShift Lightspeed Configuration

To use the generated image with the OpenShift Lightspeed operator, you need to configure an `OLSConfig` resource. Ensure that the `indexID` matches the one generated by the tool.

Example `OLSConfig` snippet:

```yaml
spec:
  ols:
    rag:
      - image: 'quay.io/myuser/acm-byok-image:2_16'
        indexID: acm_docs-2_16
        indexPath: /rag/vector_db
```

**Troubleshooting Checklist:**
1. **Architecture**: Ensure you built the image for the cluster's architecture (usually `PLATFORM=linux/amd64`).
2. **Index ID**: The `indexID` in the CR must match the `index-id` in the image's `/rag/vector_db/metadata.json`.
3. **Index Path**: The `indexPath` should always be `/rag/vector_db` as the tool hardcodes this internal path.
